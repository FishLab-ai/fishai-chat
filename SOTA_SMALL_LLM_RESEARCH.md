## 2. Key Techniques That Make Small Models Smart

### 2.1 Knowledge Distillation

**What it is**: Training a small "student" model to mimic the outputs/logits of a larger "teacher" model.

**How it's used**:
- **Llama-3.2-1B**: Derived by pruning Llama-3.1-8B, then trained with distillation on 9T tokens
- **Gemma 2-2B/9B**: Explicitly used KD to train smaller models to mimic larger teachers
- **Phi series**: Uses GPT-4-generated synthetic data as implicit distillation
- **DeepSeek-R1-Distill**: Direct reasoning distillation from the R1 model

**Effectiveness**: KD can give a 1.5B model the reasoning patterns of a 70B+ model, but domain-specific -- distilled models excel at the distribution of the teacher but may not generalize.

### 2.2 High-Quality Data Curation

**The single most important factor for small model performance.**

#### Data Mix Ratios That Work Best

| Source | Web Data | Code | Math | Synthetic |
|--------|----------|------|------|-----------|
| **SmolLM2 (Stage 1)** | 60% FineWeb-Edu + 40% DCLM (90%) | 10% StarCoderData | -- | -- |
| **SmolLM2 (Stage 2)** | 75% (60/40) | 20% | 5% | -- |
| **SmolLM2 (Stage 3)** | ~70% (40/60) | ~15% Stack-Edu | ~10% | -- |
| **Phi-3** | Synthetic "textbook" data | Significant | Significant | ~80%+ |
| **Qwen2.5** | 18T tokens, enhanced | Enhanced code data | Enhanced math data | Some |
| **SmolLM Tokenizer** | 70% FineWeb-Edu | 5% StarCoderData | 8% OpenWebMath | 15% Cosmopedia-v2 + 2% StackOverflow |

**Key findings**:
1. **FineWeb-Edu vs DCLM**: FineWeb-Edu (classifier-filtered for educational content) wins on MMLU, ARC, OpenBookQA. DCLM (instruction-following filtered) wins on HellaSwag, CommonsenseQA. Best: **blend both**.
2. **Code data matters enormously**: 10-20% code in training mixture significantly boosts reasoning capabilities even on non-code tasks.
3. **Math data needs to be added gradually**: Early introduction can cause training instability. SmolLM2 adds it in stages 2-3.
4. **Synthetic data is transformative**: Phi models are ~80%+ synthetic data. Cosmopedia (synthetic textbooks) provides high-quality educational content.
5. **Data quality > data quantity**: Qwen2 found that 7T high-quality tokens outperformed 12T lower-quality tokens for larger models.

#### Classifier-Based Filtering

The two most effective approaches:
- **FineWeb-Edu**: Llama3-70B-Instruct annotates web pages for "educational value," then a classifier filters at scale (1.3T tokens)
- **DCLM**: fastText classifier trained on OpenHermes 2.5 + r/ExplainLikeImFive data (3.8T tokens)

### 2.3 Overtraining: Beyond Chinchilla-Optimal

**Chinchilla scaling laws** suggest ~20 tokens per parameter is compute-optimal. But small models universally **overtrain** far beyond this:

| Model | Parameters | Training Tokens | Tokens/Param | Chinchilla-Optimal? |
|-------|-----------|----------------|-------------|-------------------|
| Chinchilla (theory) | 1B | ~20B | 20x | Yes |
| **Qwen2-0.5B** | 0.5B | 12T | **24,000x** | No - 1200x over |
| **Qwen2.5-0.5B** | 0.5B | 18T | **36,000x** | No - 1800x over |
| **SmolLM2-1.7B** | 1.7B | 11T | **6,500x** | No - 325x over |
| **Llama-3.2-1B** | 1.2B | 9T | **7,500x** | No - 375x over |
| **Phi-3-mini** | 3.8B | 3.3T | **870x** | No - 43x over |
| **TinyLlama** | 1.1B | 3T | **2,700x** | No - 135x over |

**Why overtraining works**: For inference-efficient deployment, you want the smallest possible model that achieves target performance. Overtraining a smaller model is cheaper at inference time than using a Chinchilla-optimal larger model with the same capabilities.

**Revised scaling law** (from SLM survey): L(N, D, Q) = E + A·N^α + B·(D·Q)^β, where Q represents data quality. This captures the fact that high-quality data changes the effective compute-optimal frontier.

### 2.4 Architecture Improvements

#### Grouped Query Attention (GQA)
- **What**: Shares KV heads across multiple query heads, reducing KV cache size
- **Adopted by**: Qwen2.5 (2 KV heads for 0.5B/1.5B/3B), Llama 3.2, Mistral, Gemma 2
- **Impact**: 2-4x reduction in KV cache memory with minimal quality loss. Critical for long-context inference on small models.

#### SwiGLU Activation
- **What**: Swish-Gated Linear Unit, replaces ReLU in FFN layers
- **Adopted by**: Virtually all modern LLMs (Qwen, Llama, SmolLM2, etc.)
- **Impact**: ~1-2% improvement in downstream tasks vs ReLU

#### RoPE (Rotary Positional Embeddings)
- **What**: Encodes position through rotation matrices in attention
- **Adopted by**: All modern LLMs
- **Impact**: Better length extrapolation, supports context extension via θ scaling (SmolLM2 uses RoPE θ=130k for 8K context)

#### Tied Embeddings
- **What**: Sharing input/output embedding weights
- **Adopted by**: Qwen2.5-0.5B/1.5B/3B, SmolLM2
- **Impact**: Saves parameters in small models (critical when embedding matrix is a large fraction of total params)

#### Multi-Stage Training
- **What**: Adjusting data mixture and learning rate across training stages
- **Pioneered by**: SmolLM2 (4 stages + context extension)
- **Impact**: Allows progressive introduction of specialized data (math, code) while maintaining stability

### 2.5 Post-Training: SFT and Alignment

#### Supervised Fine-Tuning (SFT)
- **SmolLM2**: Uses new SmolTalk dataset (conversational + task-specific + math data)
  - MagPie-Ultra (3-turn conversations from Llama-3.1-405B)
  - Smol-Constraint, Smol-Rewrite, Smol-Summarization
- **Phi-3**: Extensive SFT on GPT-4-generated instruction data
- **Qwen2.5**: Enhanced SFT with focus on structured output (JSON), long-form generation, instruction following

#### Direct Preference Optimization (DPO)
- **SmolLM2**: DPO with UltraFeedback, 2 epochs, proved most effective across benchmarks
- **Zephyr 7B**: DPO with UltraChat + UltraFeedback, outperforms Mistral base
- **Impact**: 2-5% improvement on IFEval and human preference metrics

#### RLHF
- **Llama 3.2**: Multi-stage SFT + RLHF on Meta's Research Super Cluster
- **Impact**: Significant improvement in helpfulness and safety, but expensive

### 2.6 Quantization

#### 4-bit Quantization (Q4_K_M GGUF)

**Approximate file sizes for small models**:

| Model | FP16 Size | Q4_K_M Size | Compression Ratio |
|-------|----------|------------|------------------|
| Qwen2.5-0.5B | ~1.0 GB | **~400 MB** | ~2.5x |
| Qwen2.5-1.5B | ~3.0 GB | **~1.1 GB** | ~2.7x |
| SmolLM2-1.7B | ~3.4 GB | **~1.2 GB** | ~2.8x |
| Llama-3.2-1B | ~2.5 GB | **~900 MB** | ~2.8x |

**Performance degradation from 4-bit quantization**:
- **General knowledge (MMLU)**: ~1-3% drop
- **Reasoning (GSM8K, MATH)**: ~2-5% drop
- **Coding (HumanEval)**: ~3-7% drop (more sensitive)
- **Smaller models are more affected**: A 0.5B model loses more than a 7B model at the same quantization level

**For a ~12MB 4-bit quantized model**: This would be approximately a **25-30M parameter model** (roughly SmolLM2-135M range when aggressively quantized). Expect:
- MMLU: ~10-20% (barely above random for 57 subjects)
- Any structured reasoning: Near zero
- Best use case: Simple text completion, basic chat, keyword extraction

### 3.3 Benchmark Caveats

1. **Data contamination**: Models trained on web data may have seen benchmark data. MMLU-Pro and MMLU-redux are designed to be harder/less contaminated versions.
2. **Metric artifacts**: Apparent "emergent abilities" may be artifacts of nonlinear metrics (exact-match) rather than genuine capability transitions.
3. **Few-shot sensitivity**: Small models are more sensitive to prompt format and number of shots.
4. **Instruction tuning inflates scores**: Instruct models score much higher on some benchmarks not because they "know more" but because they're better at following the benchmark's expected format.

## 4. Reaching "大模型比肩" (Comparable to Large Models)

### 4.1 Where Small Models Can Actually Compete

| Task/Benchmark | Small Model (1.5B) | Large Model (72B) | Gap | Can Compete? |
|---------------|-------------------|-------------------|-----|-------------|
| **HellaSwag** | 67.9 | ~87 | ~20 pts | Partial - good enough for many uses |
| **ARC-C** | 54.7 | ~75 | ~20 pts | Partial |
| **PIQA** | 76.1 | ~90 | ~14 pts | Yes - diminishing returns |
| **CommonsenseQA** | 34.1 | ~75 | ~41 pts | No |
| **Winogrande** | 65.0 | ~85 | ~20 pts | Partial |
| **MMLU** | 60.9 | 86.1 | ~25 pts | No - but useful for narrow domains |
| **GSM8K** | 68.5 (base) / 73.2 (instruct) | ~95 | ~22-27 pts | Getting closer |
| **HumanEval** | 37.2 (base) / 61.6 (instruct) | ~85 | ~24-48 pts | Instruct closing gap |
| **GPQA** | 24.2 | ~55 | ~31 pts | No |

**Small models can compete on**:
- Commonsense reasoning (PIQA, HellaSwag) -- diminishing returns at scale
- Instruction following (IFEval) -- SmolLM2-1.7B-I (56.7) beats Qwen2.5-1.5B-I (47.4)
- Basic text understanding and completion
- Domain-specific tasks after fine-tuning

**Small models cannot compete on**:
- Broad knowledge recall (MMLU, TriviaQA)
- Complex reasoning (GPQA, BBH)
- Competition-level math (MATH)
- Complex code generation

### 4.2 The Effective Size Phenomenon

From the SLM survey, small models can punch far above their parameter count:

| Model | Actual Size | "Effective Size" | How |
|-------|-----------|-----------------|-----|
| **Mistral-7B** | 7B | ~38B | Outperforms Llama2 on most benchmarks |
| **Phi-1 (1.3B)** | 1.3B | ~175B | Outperforms Codex-12B on HumanEval |

**The "effective size multiplier" for well-trained small models is typically 2-5x**, meaning a well-trained 1.5B model can match a poorly-trained 3-7B model.

### 4.3 Realistic Expectations for a ~12MB 4-bit Quantized Model

A 12MB 4-bit model ≈ **24M parameters** (at 4 bits/param + overhead). This is well below even SmolLM2-135M.

| Capability | Expected Performance |
|-----------|---------------------|
| **Simple text completion** | Functional for basic autocomplete |
| **Basic chat** | Very limited, often incoherent |
| **MMLU** | ~5-15% (near random for many subjects) |
| **GSM8K** | ~0-5% |
| **HumanEval** | ~0-5% |
| **Simple classification** | Possible with fine-tuning |
| **Keyword extraction** | Possible |
| **Short text generation** | Limited but functional |

**Realistic alternatives at small sizes**:
- **~50MB (Q4)**: ~100M params → SmolLM2-135M level, basic capabilities
- **~400MB (Q4)**: ~500M params → Qwen2.5-0.5B level, surprisingly capable
- **~1.1GB (Q4)**: ~1.5B params → Qwen2.5-1.5B level, genuinely useful

### 4.4 Closing the Gap: Strategies

1. **Overtrain aggressively**: 1000-3000x tokens per parameter is now standard for small models
2. **Curate data meticulously**: FineWeb-Edu + DCLM blending, staged data introduction
3. **Use synthetic data**: GPT-4/LLaMA-generated textbooks, code, and math problems
4. **Distill from larger models**: Either explicit KD or implicit via synthetic data
5. **Multi-stage training**: Adjust data mixtures based on benchmark monitoring
6. **Post-train aggressively**: SFT + DPO/RLHF dramatically improves usability
7. **Architecture optimizations**: GQA for KV efficiency, RoPE for length, SwiGLU for quality
8. **Domain specialization**: A fine-tuned 0.5B model can beat a general 7B model on specific tasks
